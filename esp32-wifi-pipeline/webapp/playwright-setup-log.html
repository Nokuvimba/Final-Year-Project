03 Errors Encountered & How They Were Solved

Seven distinct errors were encountered across three test runs. Each is documented below with the exact error message, root cause analysis, and the code change that fixed it.

Error 1 — Page title mismatch FIXED
Error message
Error: Unexpected title: "Create Next App"
Expected: true  Received: false

Root cause

The test used page.title() and checked for "MSSIA" or "Multi-Sensor". However, the Next.js root layout.tsx still had the default scaffold title "Create Next App". The title had never been updated during development.

Initial approach (rejected by student)

The first fix updated layout.tsx to set a proper title. However, the student pointed out they did not want to change the deployed app just to satisfy a test — the page's visual identity (MSSIA logo, hero heading) was what mattered.

Final fix
The test was rewritten to check visible on-screen content instead of the invisible <title> element: 
// Before
await expect(page).toHaveTitle(/MSSIA/);

// After
await expect(page.getByText("Multi-Sensor",      { exact: true })).toBeVisible();
await expect(page.getByText("Spatial Intelligence", { exact: true })).toBeVisible();
The layout.tsx was left unchanged. The test now accurately reflects what a user actually sees on screen.
Error 2 — Wrong credentials test: strict mode violation on [role="alert"] FIXED
Error message
strict mode violation: locator('[role="alert"]') resolved to 2 elements:
1) <div role="alert" class="error-msg">…</div>
2) <div role="alert" aria-live="assertive" id="__next-route-announcer__"></div>

Root cause

Next.js injects a hidden <div role="alert" aria-live="assertive" id="__next-route-announcer__"> element into every page for accessibility (screen-reader page transition announcements). This element is always present in the DOM and also carries role="alert", so Playwright's strict mode refused to act on an ambiguous match.

Fix
Changed the selector from the generic ARIA attribute to the specific CSS class used by the login form's error element: 
// Before
const errorAlert = page.locator('[role="alert"]');
await expect(errorAlert).not.toHaveText("");

// After
const errorAlert = page.locator('.error-msg');   // login form's own class
await expect(errorAlert).not.toBeEmpty();
Error 3 — Admin studio canvas not found immediately after login FIXED
Error message
Expected canvas or floor plan image in studio
expect(received).toBe(expected) → Expected: true  Received: false

Root cause

The original test used page.waitForLoadState("networkidle") then immediately called page.locator("canvas").count(). The admin studio auto-selects the first building and floor plan via React useEffect hooks — these run asynchronously after the page renders. By the time .count() was called, React had not yet completed the async data fetch and re-render that produces the canvas.

Fix
Replaced the synchronous .count() check with Playwright's toBeVisible() + a 20-second timeout, using the .or() combinator to accept either a canvas or a floor plan image: 
// Before
await page.waitForLoadState("networkidle");
const hasCanvas   = await page.locator("canvas").count() > 0;
const hasFloorImg = await page.locator('img[alt="Floor plan"]').count() > 0;
expect(hasCanvas || hasFloorImg).toBe(true);

// After
await expect(
  page.locator("canvas").or(page.locator('img[alt="Floor plan"]')).first()
).toBeVisible({ timeout: 20_000 });
Error 4 — Breadcrumb/search input not found after login FIXED
Error message
expect(locator).toBeVisible() failed
Locator: locator('input[placeholder*="Search"]')
Error: element(s) not found

Root cause

The admin studio's building search input has a dynamic placeholder: selectedBuilding?.name ?? "Search buildings…". On load, the studio auto-selects the first building ("Homes"), so the placeholder immediately becomes "Homes" — not "Search". The selector input[placeholder*="Search"] therefore found nothing.

Fix
The test was rewritten to assert elements that are always unconditionally present in the studio top bar regardless of state: 
// Before — fragile: placeholder changes after building auto-select
const searchInput = page.locator('input[placeholder*="Search"]');
await expect(searchInput).toBeVisible({ timeout: 10_000 });

// After — stable: Logout button and ADMIN badge are always rendered
await expect(page.getByRole("button", { name: "Logout" })).toBeVisible({ timeout: 15_000 });
await expect(page.getByText("ADMIN", { exact: true })).toBeVisible();
Error 5 — User viewer tests timing out at 30 seconds FIXED
Error message
Test timeout of 30000ms exceeded.
Error: page.waitForLoadState: Test timeout of 30000ms exceeded.

Root cause

All user viewer tests called page.waitForLoadState("networkidle"). This state fires only when there are no active network connections for 500 ms. Two things prevented it from ever firing:

  1. SSE (Server-Sent Events): Once a floor plan is selected, UserHeatmapViewer opens an EventSource connection to /events/floorplan/{id}/heatmap. This is a persistent, never-closing HTTP connection — networkidle can never be reached once it is open.
  2. Slow/unreachable backend: When the backend API at mssia.duckdns.org was temporarily unreachable, the fetch("/buildings") call on page load hung in-flight indefinitely, also preventing networkidle.
Fix
Replaced networkidle with load throughout the user viewer tests. The load event fires after HTML, CSS, and images are parsed — it does not wait for XHR or SSE. Element visibility assertions with generous timeouts handle the async data loading: 
// Before
await page.waitForLoadState("networkidle");

// After
await page.waitForLoadState("load");   // fast; ignores open fetch/SSE connections
A helper function gotoUserPage() was also introduced to avoid repeating this pattern in every test.
Error 6 — "Spatial Intelligence" matches two elements FIXED
Error message
strict mode violation: getByText('Spatial Intelligence') resolved to 2 elements:
1) <div class="hero-eyebrow">…</div> aka getByText('Spatial Intelligence Platform')
2) <span class="title-line title-accent">Spatial Intelligence</span>

Root cause

The home page has two elements containing "Spatial Intelligence": the hero eyebrow label reads "Spatial Intelligence Platform", and the main heading span reads exactly "Spatial Intelligence". Playwright's getByText() performs a substring match by default, so it found both.

Fix
Added { exact: true } to match only the element whose entire text content equals the string exactly: 
// Before
await expect(page.getByText("Spatial Intelligence")).toBeVisible();

// After
await expect(page.getByText("Spatial Intelligence", { exact: true })).toBeVisible();
Error 7 — "ADMIN" matches two elements (route announcer) FIXED
Error message
strict mode violation: getByText('ADMIN') resolved to 2 elements:
1) <span>ADMIN</span>
2) <div role="alert" aria-live="assertive" id="__next-route-announcer__">Map Studio — MSSIA Admin</div>

Root cause

The same Next.js hidden route announcer from Error 2 appears again. On the admin studio page it contains the text "Map Studio — MSSIA Admin" (the page title). Since this string contains "ADMIN", Playwright's substring getByText("ADMIN") matched it as well as the actual visible badge <span>ADMIN</span>.

Fix
Same fix as Error 6 — { exact: true } forces an exact full-text match, selecting only the badge <span>: 
// Before
await expect(page.getByText("ADMIN")).toBeVisible();

// After
await expect(page.getByText("ADMIN", { exact: true })).toBeVisible();

05 Key Learnings

LearningLesson
networkidle is unsuitable for SSE/streaming apps Pages that open persistent EventSource connections never reach networkidle. Use waitForLoadState("load") and rely on element timeouts instead.
Next.js injects hidden ARIA elements The #__next-route-announcer__ element carries role="alert" and contains the page title. Selectors must be specific enough to exclude it — use CSS classes or { exact: true }.
getByText() is a substring match by default Playwright's getByText("foo") matches any element containing "foo". Use { exact: true } when the text appears as a substring elsewhere on the page.
Dynamic placeholders break selectors React state can change element attributes (like placeholder) after load. Test against static, unconditional attributes or text content.
Test the UI not the document metadata The <title> tag is invisible to users. Testing the visible heading text is more meaningful and resilient to metadata changes.
Use .or() + toBeVisible() for async content When multiple elements could satisfy a condition (canvas or image), Playwright's locator.or() + toBeVisible({ timeout }) is more reliable than checking .count() synchronously.
